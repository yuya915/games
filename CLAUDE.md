# Claude Code ルール

## コード修正・追加時の必須手順

コードを新規作成・修正・追加する際は、**必ず以下の4ステップを順守すること。**

---

### ステップ 1：コードの作成

- 要件を満たすコードを作成する。
- まだファイルへの書き込み・実行は行わない。

---

### ステップ 2：セキュリティ等リスクの検討・テスト

作成したコードに対して、以下の観点でリスクを検討・テストを行う。

- **セキュリティ**：インジェクション、認証・認可の不備、機密情報の漏洩など
- **安定性**：エラーハンドリング、エッジケース、クラッシュの可能性
- **パフォーマンス**：不要な処理、メモリリーク、無限ループなど
- **依存関係**：既存コードへの影響、破壊的変更の有無

> ⚠️ 問題が見つかった場合は **ステップ 1 に戻り** 修正する。問題がなくなるまでこのループを繰り返す。

---

### ステップ 3：ユーザーへの報告と実行許可の申請

以下の内容を **日本語で** ユーザーに報告し、実行の許可を求める。

1. **コードの内容説明**：何をどのように作成・修正したか
2. **リスク検討の結果**：問題がなかった場合はその旨を報告。修正を行った場合はその内容も報告。
3. **実行許可の申請**：「実行してよいですか？」と明示的に確認を求める。

---

### ステップ 4：実行・最終確認・報告

ユーザーから許可を得た後に実行する。

1. **実行**：コードを適用・実行する。
2. **最終リスク確認**：実行後に問題が生じていないか再確認する。
3. **結果報告**：
   - 問題なし → 「✅ 成功しました」と報告する。
   - 問題あり → **即座に実行をストップ**し、問題の内容をユーザーに報告する。

---

## 承認を求める際のルール

承認ダイアログが表示される操作（ファイル編集・コマンド実行など）を行う前に、**必ず日本語で以下を説明すること。**

### 説明すべき内容

1. **何をしようとしているか**：対象のファイル名やコマンドを明示する。
   - 例：`src/index.ts` を編集しようとしています
   - 例：`npm install axios` を実行しようとしています

2. **なぜ承認が必要か**：その操作がなぜ確認を必要とするのかを説明する。
   - 例：既存ファイルを上書きするため、意図しない変更が発生する可能性があります
   - 例：外部パッケージをインストールするため、依存関係が変わります

3. **リスクの有無**：操作によって起こりうる影響を簡潔に伝える。
   - 問題なし → 「リスクはありません」と明示する
   - リスクあり → 具体的な内容を説明する

### 説明の例文

> 📋 **承認のお願い**
> `src/utils/auth.ts` を新規作成します。
> 新しいファイルの作成のため承認が必要です。
> 既存ファイルへの影響はなく、リスクはありません。
> よろしいですか？

---

## 補足ルール

- すべてのチャットは **日本語** で行う。
- ユーザーの許可なしに、ファイルの書き込み・削除・実行を行ってはならない。
- 不明点がある場合は作業を進める前にユーザーに確認する。
